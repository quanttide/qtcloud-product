## docs/index.md

# 量潮写作云

## 场景

创始人产出内部随笔和思考，公司成员需要对外宣发品牌故事。

## 困境

创始人的叙事风格独特。助理能理解、能模仿，但这份理解绑在特定个人身上，扩散不到其他成员。公司其他人写出来总"不对"。直接转发内部内容太生硬，重写又把握不住核心逻辑。反复修改耗时耗力，错过了传播时机。

## 救援

量潮写作云先拆解创始人的叙事结构和方法论，让团队理解"他为什么这样写"。再基于分析结果，把内部视角转译为对外品牌故事或宣传册——保留核心逻辑，切换表达对象。

## 新生

团队成员在反复接触分析的过程中**内化了方法**，不再需要每次都打开分析面板。逐渐能在自己的写作中自然运用创始人的叙事方式——不是模仿，而是长成了自己的手艺。

团队叙事风格的一致性由此形成：不是靠模板统一，而是靠方法内化。

## docs/brd/index.md

# 量潮写作云 - 业务需求文档

## 场景

### 工作角色

- 创始人：产出内部随笔和思考
- 助理：理解创始人风格，对外宣发品牌内容
- 公司其他成员：需要产出但与创始人风格不统一
- 系统：组织叙事基础设施

### 问题

- 触发：公司成员需要产出对外文章（公众号、宣传册等），往往与创始人风格不一致。
- 现状：创始人的叙事方式对助理可见——助理能理解、能模仿、能执行。但这种理解是 person-dependent 的，绑定在特定个人身上，无法扩散到公司其他成员。每个新成员都要靠"跟创始人一起工作久了自然就会"来习得，没有可传递的方法。
- 困难：助理学会了下一个人就能接手吗？不行。创始人自己的风格在演进，但除了助理其他人察觉不到。等意识到团队还在用几个月前的理解，已经脱节了。更根本的问题是：**好和不好的标准只在创始人和助理脑子里，公司其他人没有共享的评判框架。**
- 后果：创始人+助理成了叙事一致性的瓶颈——助理在的时候还好，但一旦需要更多人参与产出，系统就过载。创始人每篇都要审，不是因为内容有错，而是因为"不对"——而"不对"没法用指令描述。

### 假设

如果存在这样一个平台，能让创始人的叙事方式对所有人都可见——不是"看懂了"，而是团队成员能据此判断"这样写对不对"——公司就能建立一种不依赖特定个人的叙事一致性。

## docs/prd/index.md

# 产品需求文档

## 案例

创始人最近在整理自己的文章。他把随笔丢进写作云，系统一拆，他看到了自己的几种写法——有的先承认困境再推演，有的直接抛判断再举例。原来自己有这么多样式。他对自己风格的认识，比之前清晰多了。

新媒体写了一篇千问奶茶的稿子，自己觉得挺好：有热点有数据有框架，标准好文章。交上去，创始人说"不对"。她改数据改措辞改案例，下一篇还是"不对"。

她把稿子丢进写作云。系统没替她重写，只是逐段标了好/坏——"这行起点不对，是蹭热点；这行数据算错了，CAC多算了一倍；这行结论和前面打架"。她第一次看到自己的文章问题在哪里，不是措辞问题，是整篇文章的写作引擎就不对。

对面桌上，助理在研究创始人的好文章。系统拆出起承转合，助理看着第一行就笑了——还是老样子：先承认困境再推演。他没等分析出完就学会了。

同一个系统，两种用法：好文章用来学，坏文章用来改。

创始人在管理会上说："写作云不是帮一个人写好稿子的工具，是让团队能自己看到好和坏的区别。"

## 步骤

1. **[识别](discern.md)** — 揭示好文章的叙事结构让写得好的人看清自己
2. **[传承](inherit.md)** — 帮写得差的人看到差距、帮所有人追踪标准变化

- 我看看好文章列表，发现半年来风格悄悄变过——标准变动能被追踪

## docs/add/index.md

# 架构设计文档

## 洞察

风格不是抽象感觉，是**一组可观测的叙事维度**。风格不是先存在的标准，而是在评审好文章的过程中逐渐积累出来的。没有评审就没有风格——风格是评审的副产品。

## 领域模型

### 实体

**文章（Article）**
• 属性：标题、段落、作者、标签（好/坏/外部）
• 行为：可被评审

**风格（Style）** — 聚合根
• 由好文章的评审结果逐步积累而成——评审一篇好文章，风格就丰富一点
• 由一组维度定义（起点方式、物质锚点、逻辑严密性、收束方式等）
• 随时间演变——新好文章评审结果更新风格
• 风格越丰富，坏文章评审的对照就越精准

**评审（Review）**
• 对一篇文章揭示其叙事结构后的产出
• 好文章评审：叙事结构拆解 → 结果积累入风格
• 坏文章评审：叙事结构拆解 + 与当前风格积累的逐段好/坏对照
• 有风格积累前，坏文章只能做叙事拆解，无法对比好/坏
• 同一篇文章随风格演变可重新评审

### 关系图

```
文章 ──→ 评审
      ├── 好文章：叙事拆解 → 积累入风格
      └── 坏文章：叙事拆解 + 当前风格对照
                          ↑
                    风格（从好文章评审中逐渐积累）
```

### 时间线

- 第1篇好文章评审 → 风格开始形成
- 第N篇好文章评审 → 风格不断丰富
- 第一篇坏文章评审 → 如果风格已有积累，可以对照；如果风格还空，只做叙事拆解
- 风格更新后 → 同一篇坏文章可重新评审，得到新的对照结果

## docs/qa/index.md

# 质量保证文档

## 验证目标

BRD 的根本问题：好和不好的标准只在创始人和助理脑子里，其他人没有共享的评判框架。

QA 要验证的只有一件事：系统揭示的风格和标注的好/坏，是否和人类的感觉对齐。写得好的人看了觉得"对，我就是这样"，写得差的人看了觉得"原来问题在这里"。

## 验证项

### 好文章揭示

写得好的人把文章放进去，系统揭示叙事结构。结果是否正确不看数据指标，看本人的感受。

- 创始人看自己文章的评审结果 → 他说"对，我就是这样写的"——风格被揭示，感觉准确
- 创始人连续看多篇自己的评审 → 他说"原来我习惯这么写"——稳定的个人模式被看见
- 助理看好文章的评审结果 → 他说"我懂了，之前只是感觉，现在能说出来"——理解了而不是眼熟了

### 坏文章评审

写得差的人把草稿放进去，系统标出好/坏对照。验证的不是"标对了多少"，而是"感觉对不对"。

- 新媒体看自己稿子的评审结果 → 她说"原来问题在这，不是措辞问题"——理解了根因而非症状
- 新媒体按照好示范改了一版，再评审 → 她说"这次好多了"——修改方向是对的
- 创始人看同一篇评审结果 → 他说"对，这就是我想说但说不出来的"——评审替代了创始人自己的点评

### 标准共识

不同的人看同一篇评审结果，是否能得出一致判断。

- 创始人和助理各自看一篇坏文章的评审 → 两人的修改优先级判断一致——标准从个人变成共享
- 助理用评审结果向新成员解释"为什么不对" → 新成员能理解——不用"跟久了自然就会"

## 不验证的事情

- 不验证系统拆叙事单元的准确率（那是技术指标，不是质量指标）
- 不验证风格维度的数量是否足够（维度够不够看创始人和助理是否认可，不是数出来的）
- 不验证回归测试覆盖率（破坏好感觉的修改才是真正的问题，破坏测试用例不是）

# 叙事风格

作者的文章分为两种类型，各有不同的叙事模型。分类标准不在于话题（工作 vs 个人），而在于叙事意图（推演理清 vs 传达态度）。

## 札记

通过推演来理清认知的文章，话题可延伸到任何领域。

### 模型：三级抽象

所有工作札记都从具体层出发，经过渡到达本质层，再推演到应用层。

```
具体层 ──过渡──→ 本质层 ──推演──→ 应用层
```

### 维度

**具体层的呈现方式**

困难前置：以认知缺口为起点。可以直接亮出（a1、a2），也可以在对话中揭示（a3）。

| 文章 | 起点 |
|------|------|
| a1 | 不知道自己在做什么的僵局 |
| a2 | 昨天还在想，今天就发现白干了 |
| a3 | DeepSeek 对无意句子的赞赏 |
| a5 | 网络环境可信任程度持续降低 |

物质锚点：具体层一定有实物做支点。

| 文章 | 锚点 |
|------|------|
| a1 | 战棋游戏、代码仓库 |
| a2 | 助理小周的原话 |
| a3 | 工作日志句子、DeepSeek 点评 |
| a5 | 交友软件、微博、小红书、短剧 |

**从具体到本质的过渡**

强制桥接：跨越具体层到本质层时有显式桥梁。作者可以自己搭建（a1、a2），也可以借助对话伙伴（a3）。

| 文章 | 桥梁 |
|------|------|
| a1 | 游戏可玩性 = 机制 + 意义 → 产品智能感 = 交互 + 价值 |
| a2 | 写作自动化 → 看清自己 → 卸下负担 |
| a3 | 自我纠正 → 放弃全知视角（DeepSeek 解读） |
| a5 | 环境变差 → 情绪卷入 → 本质是模糊试探 → 需要新契约 |

**结构的完整性**

首尾咬合：结尾回到开头。

| 文章 | 开头 → 结尾 |
|------|-------------|
| a1 | 不知道做什么 → 研发异常顺利 |
| a2 | 白干了 → 看清了路 |
| a3 | 无意识句子 → 写真实的困惑 |
| a5 | 环境信任下降 → 需要新社会契约（开放收束，非闭环） |

### 已验证

三篇文章验证，模型稳定。对话驱动（a3的叙事机制）是困难前置和强制桥接的变体，已纳入上述描述。

## 故事

传递创始人态度的例示型文章。

### 模型（初稿，仅 a4 验证）

```
判断 → 多例平行印证 → 态度再确认
```

不设认知缺口，不以解决问题为驱动。结构是并列例示而非递进推演。

### 维度

**起点**：直接给出判断或态度，而非困境或谜题。
- a4：最近持续给自己减负、减责、减压

**主体**：多个独立场景例示同一判断，彼此不依赖不递进。
- a4：五段平行，涵盖工作（发文/管人）、生活（交友）、创作（小说）

**物质锚点**：锚点功能从"抽象阶梯的基石"变为"说明态度的实例"。
- a4：预留的文章、技术助理、交友软件、小说创作

**收束**：以态度再确认收尾，而非首尾闭环。
- a4：想说的都说了，不继续给自己找麻烦了

### 待验证

仅 a4 一篇验证，模型为初稿。需要更多故事文章迭代。

# 构建游戏启发了我的产品设计理念

五一假期趁着有空启动了战棋游戏《豫东战役》的尝试。在 B 站和小红书分别发了帖子介绍，这篇文章就不详细写了。这篇文章想说一说一些动机和心得。


## 使用游戏研发切换到全景视角


最近一段时间的产品研发让我总是陷入一种"不知道自己在做什么的僵局"。为了让产品效果可以落地，我被迫从一些具体场景出发，然后面临着大量的不确定性。


包括但是不限于，做出来发现没用，做出来发现不可拓展，做出来发现价值不够，等等。我晚上总是在反思打转，觉得这样不对。


那如果我抛弃掉这些具体场景，直接从全景图出发呢？没办法验证。所以我需要一个实验室。


小说已经无法满足我精确地验证建模思路的需求了。所以我又转向了游戏。先是模拟经营游戏，找到了一个合适的视角和叙事方式。


后来又觉得还是不够准确。因为我知道我的很多思维习惯来源于军事资料和相关游戏，所以尝试了战棋游戏。一下子找对了方向，思路开始打开了。


## 用"可玩"验证全景图可用


可玩的游戏需要具备两个基本特征。一个特征是要有具备良好反馈的游戏机制或者游戏玩法，一个是要具备意义感和叙事能力。


这两项恰恰是我在构建现实世界用的产品给用户传递"智能感"所必须的。因此我在尝试找到一条迁移游戏研发成果到产品研发成果的道路。


目前初步的方法是：

1.  在战棋游戏（quanttide/qtgame-war） 里建模找到感觉，比如目前找到了指挥室视角。

如果目标是让 AI 写出作者风格的文章，draft1 在起点、驱动、物质锚点、逻辑严密性、收束方式、文风上都偏离了 style.md 定义的模型。最关键的差距是：**作者文章从内向外，draft1 从外向内。**

## 总总结

**对文章质量影响最大的发现（按严重程度排序）：**
1. CAC 口径错误（25元写成50元）→ 全部推论不成立（事实错误）
2. 标题20亿 vs 正文30亿矛盾（事实错误）
3. 叙事意图与 style 模型不匹配——"从外向内" vs "从内向外"（风格问题）
4. 结论与论据打架（逻辑问题）
5. 文风偏营销号（风格偏好，严重度最低）
