# 风格

## 用案例定义标准，不先列规则

先积累足够多的好案例（Article），对它们做结构化拆解（评审 Review）；随着案例增多，标准（Style）逐渐显现并迭代。同一篇坏文章可以在风格更新后重新评审，得到新对照。

### 基于案例的设计

基于案例的设计（Case-Based Design, CBD）/ 基于案例的推理（Case-Based Reasoning, CBR）：核心就是「新问题要解决时，先检索相似旧案例，复用并适配其方案；再保存新案例进库，不断积累」。经典四步：检索 → 复用 → 评审/修正 → 保留（库维护）。

在写作云里：库 = 已评审的好文章；检索/复用 = 新稿与当前风格的逐段对照；保留 = 新好文章的评审结果更新风格。

### 落地建议

- 建一个案例库（Case Base）来承载标准，而不是一次性写死评分卡
- 每次评价新稿时，从库里检索最相似/最相关的好稿作为对照，而不是对着空白规则打分
- 评价完以后，把有代表性的好稿（含评审结构）存回库，让标准逐步演变

## 在用户现场观察与访谈，把说不出来的标准挖出来

文档里反复说「好和不好的标准只在创始人和助理脑子里」；解决方式是「把他们自己也没完全显化的判断，通过评审面板暴露出来」。这和情境调查/情境设计的目标高度重合。

### 情境调查

情境调查（Contextual Inquiry）：强调在用户工作的真实环境里，边观察边访谈，把隐性工作流和知识挖出来，而不是只在会议室问需求。

情境设计（Contextual Design）会把田野数据用亲和图（Affinity Diagram）等方式结构化，再推导出系统模型。

### 落地建议

- 让评审面板站在创始人和助理的肩膀上
- 提前做几轮情境访谈/观察，问清「你觉得哪里不对、但说不清楚为什么」的场景，把这些模糊判断系统性地记录下来
- 把这些访谈洞察变成拆解维度（起点方式、物质锚点、收束方式等）的初始种子，而不是直接拍脑袋定
- 在后续版本中，把评审结果反向导出给创始人和助理看，再根据他们的体感修正——本质上就是田野数据与系统输出之间的校准循环

## 用对照来做评审与共识校准

左栏文章 / 右栏分析；好稿 vs 坏稿；同篇不同时间的评审。对照本身就是核心评价工具。

### 相关成熟方法

设计批评 / 工作室评审（Design Critique）：在设计教育与实务里，Critique 是通过「摆出作品 + 多人讨论 + 比较不同方案」来提高判断力与作品质量的方法。关键：多看样子，从差异里看结构和问题。

启发式评估/专家评审中的多评审人校准（Calibration）：通过让多位评审人独立评价相同案例，然后对齐差异，来提高一致性。UX 与内容审核领域都在用校准会来统一判断尺度。

### 落地建议

- 在团队里开对照评审会：选 2–3 篇典型稿（含好/坏），让不同成员先独立写问题清单/亮点清单；现场逐条对比，找出分歧最大的点，用创始人和助理的判断做锚点，逐步形成共享评判框架
- 把对照做成产品机制：新稿总是和近期被认可的好稿并排呈现；把差异清单（哪里不同）自动生成出来，而不是只给一个总评分

## 把好稿的共性提炼成可复用的模式

style.md 里把创始人的叙事归纳成两种模型和若干维度，并在不同案例上验证（3 篇 vs 1 篇）。这是典型的模式语言实践。

### 模式语言

模式语言（Pattern Language）：Christopher Alexander 的方法，主张从大量案例里抽取重复出现的、稳定有效的局部解决方案，编成模式集，供人组合复用；模式是从实践中长出来的，不是从理论推下来的。

### 落地建议

- 建一个轻量的模式库：每个模式写清适用场景（何时用）、结构模板（起承转合的关键动作）、常见失败方式；每个模式至少用 3 个以上真实案例验证，对用户可见地标注「已验证/待验证」
- 在产品里提供按模式写/按模式改的辅助：写之前可选适用模式；写完之后把模式结构叠在文稿上，高亮偏离点

## 两个人一起设计评审，快速对齐说不出的标准

文档里反复出现的「创始人与助理对同一篇稿的判断要一致」，本质上就是在要求两人共享同一套隐性标准。

### 结对设计

结对设计（Pair Design）：两位设计师（或设计师+工程师、评审+作者）实时协作，在同一个界面/白板上一起分析、一起取舍，用即时讨论把模糊感觉变成显式决策。

### 落地建议

- 在关键评审场景中引入结对评审：创始人 + 助理、或评审员 + 作者，共同站在评审面板前逐段讨论
- 把讨论结果（尤其是分歧 → 如何对齐）记录下来，作为风格更新的证据
- 用产品记录结对决策：对每一处改/不改，记录为什么改、参照哪篇好稿的哪一段，把决策链条显化

## 不把技术指标当成质量指标

QA 明确写「不验证准确率、不验证维度数量、不验证覆盖率」。这在 UX 与评估方法里是有成熟先例的：UX 评估方法会区分「技术层面的可靠性」（如测试覆盖、算法精度）和「体验层面的有效性」（用户是否真的更好用、更理解），并用不同方法去分别验证，而不是互相替代。

### 落地建议

- 在产品度量上设置两条线：系统层关注稳定性、延迟、安全、算法一致性；体验层关注「作者是否觉得问题被讲清楚了」「评审是否真的减少了返工次数」「团队对标准的共识是否提高」
- QA 报告拆成两份：一份给工程（技术指标），一份给业务（体验/感知指标），并写清楚「我们故意不优化什么」

## 让标准随时间演变，不一次性固化为评分卡

ADD 的时间线强调「风格随评审不断演变；同篇坏稿可重新评审」。这和 CBR 的维护/演化阶段、模式语言的持续演化是一回事。

### 落地建议

- 在产品里增加版本时间线：展示风格模型在哪些时间点被哪些好稿更新过；支持回滚到历史版本做对照
- 提醒用户「你现在的标准可能已经过期」：如果某篇稿是半年前的评审，UI 上明确提示当前风格已更新，建议重新评审

## 方法串联

如果你要继续往下做，可以直接把这些方法按阶段排成一条链：

**前期**：情境调查（Contextual Inquiry）—— 把隐性的好/坏标准从创始人和助理身上挖出来。

**中期**：
- 基于案例的设计（CBD/CBR）—— 用案例库与检索构建评价能力，而不是一次性规则
- 模式语言（Pattern Language）—— 把案例共性提炼为可复用模式

**后期**：
- 设计批评（Critique）+ 校准会（Calibration）—— 用对照/讨论在团队内形成共享标准
- 结对设计/评审（Pair Design）—— 关键决策时刻两个人一起做，快速对齐隐性判断

**度量**：把「人感觉对不对」作为质量线，同时保留必要的工程指标，但不要把两者混为一谈。

这些方法都不是似是而非的方法论，而是业界已经反复实践、有清晰步骤、能直接照着落地的方法。写作云这套文档，可以看作是 CBD + Contextual Design + Pattern Language + Critique + Calibration + Pair Design 在叙事/写作场景上的一次具体编排。
