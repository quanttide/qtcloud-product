# product-awareness

从产品文档中提取设计者的创作潜意识，输出结构化觉察报告。

## 核心定位

觉察只做一件事：从文档的决策模式中反向推导出设计者的根隐喻，而不是评论文档写得好不好。

### 三改变

- 从「写了什么」→「为什么会这样写」：从内容到驱动力
- 从「并列特征」→「层级结构」：从散点到关系
- 从「旁观描述」→「置身其中」：从外部观察者到内部结构重建

### 边界

觉察不评价、不建议、不纠错。

错误：这个决策是错的，应该这样改。
正确：这个决策反复出现，它指向设计者的一个根假设：「……」——这个假设在场景 X 下不成立，但设计者自己没有意识到。

觉察报告只呈现设计者自己可能看不到的结构和边界。

## 执行流程

### 1. 沉浸阅读

通读源文档至少两遍。第一遍理解内容，第二遍找模式。

记录时标注以下信号：
- 重复出现但没有被正面定义的概念
- 跨章节的一致性选择（同一类决策在不同文档中反复出现）
- 被排除的东西（文档明确说"不要什么"）
- 命名方式（动词 vs 名词、动作 vs 对象）

### 2. 找根

从记录中提取一条统领性的根隐喻。检验标准：根隐喻如果能自洽地解释大多数观察到的决策模式，就是对的。如果不能，继续深度阅读。

根隐喻通常是一句话，形如「X 应当 Y」或「X 就是 Y」——它是一个关于世界应当如何运作的隐含判断。

典型的根隐喻示例：
- 产品文档的设计者：「隐性的东西应当被显化，绑在人身上的东西应当成为公共的」
- （不是）"这个产品要解决写作一致性的问题"——那是产品价值主张，不是根隐喻

区分方法：
- 产品价值主张回答"产品做什么"
- 根隐喻回答"设计者认为世界应该怎样"——它在产品出现之前就存在于设计者脑中

### 3. 描述生长方式

观察根隐喻如何支配衍生决策。组织为层级结构，不是并列列表。

```
根
├── 生长方式（如何从根走到具体选择）
├── 判断方式
│   ├── 方向判据
│   └── 验证工具
└── 约束层（知道不做什么）
```

每一项必须能从根隐喻推导出来，而非独立的观察。

### 4. 寻找盲点

根隐喻有它处理不了的东西。找到它。

方法：假设根隐喻为真，遍历源文档中的所有角色和场景，检查哪个角色/场景在根隐喻的框架下找不到位置。

盲点不是"文档遗漏了什么功能"，是"设计者的认知结构里天然装不进去的东西"。

### 5. 命名觉察结果

用一个词或一个短句概括整体观察结果。这通常与盲点有关——因为觉察的价值不在于确认已知，在于发现边界。

## 标准结构

```
# 产品觉察

> <源文档路径和日期>

---

## 根

（根隐喻陈述 + 跨文档证据链）

## 如何生长

（根隐喻如何支配衍生决策，层级结构图）

## 如何判断

（方向判据 + 验证工具，均从根推导）

## 知道不做什么

（被排除的东西及其与根的关系）

## 盲点

（根隐喻处理不了的场景或角色）

## 觉察

（一个词或一句话的收束，通常与盲点有关）
```

## 验证方式

- 源文档中的跨章节一致性决策能被根隐喻解释
- 结构是层级而非并列——上层解释下层
- 盲点是根隐喻的结构性盲区，不是设计遗漏
- 全文不做评价、不写建议、不纠错

## 检验问题

- 如果把根隐喻告诉设计者，他会说"对，但你没说之前我确实没意识到自己是这样"——这是成功
- 如果他说"你总结得好"——这是失败（只做了归纳，没做觉察）
